# ClawGuard 项目技术栈说明

## 1. 项目定位

ClawGuard 当前是一个围绕 OpenClaw / Skill 生态安全治理构建的单仓全栈项目。它不只是一个前端展示面板，而是把以下几类能力放在了同一个仓库中：

- OpenClaw 公网暴露面监测与可视化
- OpenClaw 风险情报与漏洞跟踪
- 安全研究论文聚合与前沿观察
- Skill 生态静态扫描、风险画像与搜索
- 本地上传 / 仓库地址 / 技能市场下载的 Skill 扫描能力
- Node.js 前后端与 Python 扫描器协同运行

当前主业务应用在 `web` 目录下，扫描器体系集中在 `scanners` 目录下。

## 3. 总体架构

项目整体采用“前后端同仓 + 扫描器外接 + 数据库存储 + 本地缓存”的架构。

### 3.1 核心组成

1. 前端：React + Vite 的单页应用控制台
2. 后端：Express API 服务，统一提供业务接口
3. 数据层：MySQL + Prisma
4. 数据脚本层：Node 脚本负责公网暴露数据生成与导入
5. 情报抓取层：后端服务直接访问 GitHub、NVD、Crossref、DBLP、arXiv 等外部源
6. 扫描层：Node 调用 Python 扫描器完成 Skill 静态分析与深度分析
7. 缓存层：内存缓存 + 文件缓存 + 数据库快照

### 3.2 当前业务域

当前项目至少覆盖以下 4 个业务域：

- `OpenClaw Exposure`
  - 暴露 IP、地域分布、趋势、版本分布
- `OpenClaw Risk`
  - GitHub Security Advisory / NVD 漏洞聚合、修复进度、风险追踪
- `Security Research`
  - 安全研究论文和预印本检索、聚合、排名
- `Skill Governance`
  - Skill 风险统计、扫描批次、危险样本、技能搜索、交互式扫描

- Express 5

后端入口文件是：

```text

```

当前主要 API 分组如下：

#### 1. 基础接口

- `GET /api/health`

#### 2. 暴露面接口

- `GET /api/exposure/stats`
- `GET /api/exposure/world-distribution`
- `GET /api/exposure/china-distribution`
- `GET /api/exposure/trend`
- `GET /api/exposure/version-trend`
- `GET /api/exposure/list`

#### 3. OpenClaw 风险接口

#### 4. 安全研究接口

- `GET /api/security-research/overview`
- `GET /api/security-research/papers`
- `POST /api/security-research/refresh`

#### 5. Skill 治理接口

- `GET /api/skill/intelligence/overview`
- `GET /api/skill/search`

### 5.3 后端服务职责

后端不仅负责常规 CRUD / 查询，还负责：

- 定时拉取 OpenClaw 风险情报
- 定时拉取安全研究论文数据
- 在数据库中持久化快照
- 提供带缓存的聚合查询
- 在开发阶段通过 Vite 插件补充 Skill 扫描 API

### 5.4 自动调度

后端启动时会执行：

- `initializeOpenclawRiskScheduler()`
- `initializeSecurityResearchScheduler()`

这说明项目内置了情报与研究数据的自动刷新机制，而不是单纯靠手工脚本更新。

数据库使用 MySQL，ORM 使用 Prisma。

Schema 文件：

```

- 数据库类型：MySQL
- Prisma Client 输出目录：`web/generated/prisma`

### 6.2 已显式建模的 Prisma 模型

- `ExposureSnapshot`
- `ExposureRecord`
- `ExposureDailyAgg`

#### OpenClaw 风险相关

- `OpenclawRiskSnapshot`
- `OpenclawRiskIssue`

- `SecurityResearchSnapshot`

### 6.3 数据库存储特点

项目数据库并不只是业务展示表，还承担了“快照仓库”的角色：

- 风险数据按快照落库
- 研究论文按快照落库
- 暴露面数据按日期快照和聚合表存储

### 6.4 额外依赖的技能库表

当前项目还直接通过原生 SQL 查询以下表：

- `skillrecord`
- `skillstaticscanbatch`
- `skillstaticscanresult`
- `skillimportbatch`

这些表没有完整出现在当前 `schema.prisma` 中，但已经被 `skillIntelligenceService.mjs`、`skillSearchService.mjs`、`scan-skill-static.mjs` 等模块直接依赖。

因此，当前数据库可以理解为两部分：

1. Prisma 明确建模的业务数据表
2. 现有技能库导入与静态扫描结果表

## 8. Skill 扫描与多扫描器体系

这是当前项目区别于普通 React + Express 项目的一个核心点。

### 8.1 扫描器目录

项目当前存在多个扫描器版本：

- `scanner1`
  - 较早期 Skill 扫描器，脚本入口为 `scripts/scan_skill.py`
- `scanner2`
  - 统一扫描 CLI，当前在 Vite 插件和批量静态扫描脚本中被重点使用
- `scanner5`
  - 偏 MCP/Agent 化的 Python 扫描系统
- `scanner6`
  - 偏 JS 审计器原型

### 8.2 当前主用扫描器

从 `web/vite.config.js` 和 `web/server/scripts/scan-skill-static.mjs` 看，当前主流程主要依赖：

- `scanner1`
- `scanner2`

其中：

- `scanner1` 负责基础静态扫描
- `scanner2` 负责统一 CLI、扩展扫描、深度分析流程

### 8.3 Skill 扫描输入来源

当前 Skill 扫描支持三类输入：

1. 通过 slug 从技能市场下载
2. 通过 Git 仓库 URL 拉取
3. 通过前端上传文件 / zip 包

### 8.4 Node 与 Python 协作方式

在 `vite.config.js` 中，Vite 被扩展了一个自定义插件 `skillScanApiPlugin()`，它会：

- 接收前端的 Skill 扫描请求
- 把上传文件临时落盘
- 解压 zip 包
- 调用 Python 扫描器
- 返回静态报告
- 在有鉴权和 API Key 的情况下启动深度分析任务
- 提供 `/api/skill/scan/status` 查询异步任务状态

### 8.5 批量静态扫描

`web/server/scripts/scan-skill-static.mjs` 说明项目还有一条批量离线扫描流水线：

- 从 `skillrecord` 表中读取仓库
- 本地 clone 仓库到缓存目录
- 自动寻找 `SKILL.md`
- 复用扫描结果
- 写回 `skillstaticscanresult`
- 更新 `skillstaticscanbatch`

这意味着当前项目已经具备 Skill 仓库级别的批处理扫描能力。

## 10. 缓存与持久化策略

当前项目使用了多层缓存设计。

### 10.1 内存缓存

多个服务中都存在内存缓存，例如：

- 暴露面接口缓存
- OpenClaw 风险缓存
- 安全研究缓存
- Skill Intelligence 概览缓存

### 10.2 文件缓存

风险情报和研究论文会写入仓库外的缓存目录，例如：

- `clawguard-cache/openclaw-risk`
- `clawguard-cache/security-research`
- `clawguard-cache/skill-static-scan`

### 10.3 数据库快照

即使缓存失效，系统仍然可以从数据库快照恢复展示数据。

```

### 12.2 前端本地变量

当前项目使用了 `.env.local` 注入 Vite 变量，例如：

- `VITE_ADMIN_DEFAULT_API_KEY`

文档中不应记录真实密钥值，实际部署时应把它视为敏感配置。

### 12.3 风险与研究服务相关变量

从代码可见，项目还支持通过环境变量控制外部服务行为，例如：

- OpenClaw 风险仓库、关键字、缓存目录、刷新周期
- GitHub Token、NVD API Key
- Security Research 检索关键字、并发、超时、缓存目录、联系邮箱
- Skill 静态扫描缓存目录、并发度、缓存仓库数量

这说明项目已经进入“可配置服务化”阶段，而不是简单 demo。
